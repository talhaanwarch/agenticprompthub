- **Date:** 2026-06-22
- **Status:** Draft

- **Blocks:** Nothing (SDK is consumed by user applications, not subsequent backend parts)

---

## Overview

The SDK is a thin TypeScript npm package that lets Node.js applications fetch rendered prompts at runtime — without touching the database, knowing about version numbers, or managing aliases. It is the "aha moment": a developer replaces a hardcoded prompt string with a single call to `renderPrompt('my-prompt', 'production', { name: 'Alice' })` and receives an OpenAI-compatible messages array, ready to pass directly to any LLM API.

All rendering (Jinja2/nunjucks template evaluation) happens **server-side**. The SDK is intentionally thin: it sends variables to the render endpoint and returns the result. No template parsing, no nunjucks dependency.

**Node.js only.** Browser support is deferred (no use case in Phase 1; LLM calls are server-side).

**Definition of done:** a Node.js script using the SDK calls `renderPrompt("my-prompt", "production", { name: "Alice" })` and receives a correct OpenAI-compatible messages array — without touching the database directly.

### Types

/** A single message in an OpenAI-compatible chat format. */
interface Message {

## Constructor Behavior

Resolution order for each config field: **constructor arg → environment variable → default**.

| Field | Env var | Default | Throws if missing? |
|---|---|---|---|
| `apiKey` | `AGENTICHUB_API_KEY` | — | Yes — throws `MISSING_API_KEY` |
| `baseUrl` | `AGENTICHUB_BASE_URL` | — | Yes — throws `MISSING_BASE_URL` |
| `cacheTtl` | — | `60_000` | No |
| `maxCacheSize` | — | `500` | No |
| `maxRetries` | — | `1` | No |
| `retryInterval` | — | `500` | No |

**Fail fast:** config validation runs in the constructor, not at first use. A misconfigured client throws immediately — before any network calls can be made.

**`baseUrl` normalisation:** strip any trailing slash from `baseUrl` in the constructor so callers don't have to worry about it.

**Cache initialisation:** the module-level cache singleton is sized by the **first** `AgenticHub` constructor call (the `maxCacheSize` from the first instance wins). Creating multiple `AgenticHub` instances with different `maxCacheSize` values is unsupported. The expected usage pattern is to create one instance at process startup and reuse it.

## Cache Design

### Singleton cache

The cache is a **module-level singleton** — one `LRUCache` instance shared across all `AgenticHub` instances in the same Node.js process:

```typescript
// Created once when the module is first imported.
// maxSize is patched from the first AgenticHub constructor call.
let _cache: LRUCache<string, { value: Message[]; fetchedAt: number }> | null = null;

function getCache(maxCacheSize: number): LRUCache<...> {
  if (!_cache) {
    _cache = new LRUCache({ max: maxCacheSize });
  }
  return _cache;
}
```

TTL is **not** delegated to lru-cache's built-in TTL — the `fetchedAt` timestamp is stored with each entry and age is computed manually on read. This allows the stale-while-revalidate pattern (lru-cache's TTL evicts the entry on read; we want to keep it for stale serving).

### Cache key

```
${apiKey}:${promptName}:${alias}
```

Scoped to `apiKey` so that different teams (using different keys against the same server) never share cache entries.

### Stale-while-revalidate algorithm

On every `renderPrompt(name, alias, variables)` call:

1. **Compute cache key** → `${apiKey}:${name}:${alias}`
2. **Cache hit, fresh** (`age < cacheTtl`): return `entry.value` immediately. Done.
3. **Cache hit, stale** (`age >= cacheTtl`): return `entry.value` immediately AND fire a background refresh (fire-and-forget, not awaited).
4. **Cache miss**: fetch from API, store result in cache with `fetchedAt: Date.now()`, return result.

**Note:** variables are NOT part of the cache key. The cache stores the rendered output for a given prompt+alias; different variable sets produce different rendered output but the same cached entry will be returned. This is a known limitation for Phase 1 — the primary use case is stable variable sets per deployment (e.g., user name is always passed fresh per call, not expected to be cached). If variable-sensitive caching is needed later, it can be added by hashing variables into the key.

### Background refresh failure handling

If the background refresh fails (network error or 5xx):
- Log: `[AgenticHub] Background refresh failed for "${name}/${alias}" — continuing to serve stale`
- Leave the stale entry in place (it will be re-attempted on the next `renderPrompt` call)
- Do NOT throw — the refresh is fire-and-forget; the main call already returned

### Cold cache + API unreachable

No stale entry exists → must fetch → retries exhausted → throw `AgenticHubError` with code `NETWORK_ERROR`.

## Retry Logic

Retry applies to **background refreshes and foreground fetches alike**.

```typescript
async function fetchWithRetry(
  url: string,
  init: RequestInit,
  maxRetries: number,
  retryInterval: number
): Promise<Response>
```

- **Total attempts:** `1 + maxRetries` (e.g., `maxRetries=1` means 2 attempts total)
- **Retry condition:** network-level error (fetch throws) OR HTTP 5xx response
- **Do NOT retry:** HTTP 4xx responses — these are client errors (bad variables, wrong name, invalid key); retrying won't help
- **Delay:** `await sleep(retryInterval)` between attempts (simple fixed delay; no exponential backoff in Phase 1)
- **After all attempts exhausted:** rethrow the last error (wrapped in `AgenticHubError` by the caller)

```typescript
async function sleep(ms: number): Promise<void> {
  return new Promise((resolve) => setTimeout(resolve, ms));
}
```

## Error Types

```typescript
type AgenticHubErrorCode =
  | 'MISSING_API_KEY'     // Constructor: no apiKey in args or env
  | 'MISSING_BASE_URL'    // Constructor: no baseUrl in args or env
  | 'NETWORK_ERROR'       // Fetch threw / all retries exhausted
  | 'API_ERROR'           // Non-retriable HTTP error (4xx, or 5xx after retries)
  | 'MISSING_VARIABLES';  // 400 from render endpoint: template has variables not sent

class AgenticHubError extends Error {
  readonly code: AgenticHubErrorCode;
  readonly statusCode?: number;   // HTTP status code, if applicable
  readonly body?: unknown;        // Parsed response body, if applicable

  constructor(
    message: string,
    code: AgenticHubErrorCode,
    statusCode?: number,
    body?: unknown
  ) {
    super(message);
    this.name = 'AgenticHubError';
    this.code = code;
    this.statusCode = statusCode;
    this.body = body;
  }
}
```

### Error mapping

| Situation | Code | statusCode | body |
|---|---|---|---|
| No `apiKey` at construction | `MISSING_API_KEY` | — | — |
| No `baseUrl` at construction | `MISSING_BASE_URL` | — | — |
| Fetch throws (network) after retries | `NETWORK_ERROR` | — | — |
| HTTP 400 with missing variables | `MISSING_VARIABLES` | 400 | `{ missing: string[] }` |
| HTTP 401 | `API_ERROR` | 401 | response body |
| HTTP 404 (prompt not found) | `API_ERROR` | 404 | response body |
| HTTP 5xx after retries | `API_ERROR` | 5xx | response body |

For `MISSING_VARIABLES`, the `body.missing` array (returned by the API) is preserved so callers can log exactly which variables were omitted.

## HTTP Request Details

### Render request

```
POST {baseUrl}/prompts/{name}/{alias}/render
Authorization: Bearer {apiKey}
Content-Type: application/json

{ "variables": { ...variables } }
```

- `variables` defaults to `{}` if not provided by the caller
- Response 200: `{ "messages": [{ "role": "...", "content": "..." }, ...] }`
- Return `response.messages` as `Message[]`

### Response handling

```typescript
const res = await fetchWithRetry(url, { method: 'POST', ... }, maxRetries, retryInterval);

if (!res.ok) {
  const body = await res.json().catch(() => undefined);
  const code = res.status === 400 && body?.missing ? 'MISSING_VARIABLES' : 'API_ERROR';
  throw new AgenticHubError(
    `AgenticHub API error ${res.status}`,
    code,
    res.status,
    body
  );
}

const data = await res.json();
return data.messages as Message[];
```

## Testing

### Integration test (required)

One end-to-end integration test that exercises the full stack:

1. Start the local API server (or point at a test environment with `AGENTICHUB_BASE_URL`)
2. Create a prompt via `POST /prompts`
3. Commit a version with a Jinja2 template: `"Hello, {{ name }}!"` (via `POST /prompts/:id/versions`)
4. Promote `production` alias to version 1
5. Instantiate SDK: `new AgenticHub({ apiKey: testApiKey, baseUrl: testBaseUrl })`
6. Call `sdk.renderPrompt('my-prompt', 'production', { name: 'Alice' })`
7. Assert response: `[{ role: 'user', content: 'Hello, Alice!' }]`

### Unit tests (mock fetch)

| Test | How |
|---|---|
| Constructor throws `MISSING_API_KEY` when no key | Pass empty config, no env var set |
| Constructor throws `MISSING_BASE_URL` when no URL | Pass `{ apiKey: 'x' }`, no env var |
| Fresh cache hit returns without fetching | Prime cache manually; assert fetch not called |
| Stale cache returns immediately and triggers background refresh | Mock fetch to resolve after delay; assert return is instant |
| API unreachable + stale entry → return stale, log warning | Mock fetch to reject; assert stale value returned and `console.warn` called |
| Cold cache + API unreachable → throws `NETWORK_ERROR` | Mock fetch to always reject; `maxRetries=1` → 2 attempts → throw |
| Retry: 2 network failures then success → returns value | Mock fetch to reject twice, resolve on 3rd |
| 400 response with `{ missing: ['name'] }` → throws `MISSING_VARIABLES` | Mock fetch to return 400 body |
| 401 response → throws `API_ERROR` with statusCode 401 | Mock fetch to return 401 |
| 404 response → throws `API_ERROR` with statusCode 404 | Mock fetch to return 404 |
| 500 response (after retries) → throws `API_ERROR` | Mock fetch to always return 500; `maxRetries=1` → 2 attempts → throw |
| Cache key includes apiKey (two keys → no cross-contamination) | Two SDK instances, different keys; assert separate cache entries |

**Config reference**

| Option | Env var | Default | Description |
|---|---|---|---|
| `apiKey` | `AGENTICHUB_API_KEY` | required | Team or personal API key |
| `baseUrl` | `AGENTICHUB_BASE_URL` | required | AgenticHub API base URL |
| `cacheTtl` | — | `60000` (60s) | How long to serve a cached result before background-refreshing |
| `maxCacheSize` | — | `500` | Max prompt entries in memory |
| `maxRetries` | — | `1` | Retries on transient failure (2 total attempts) |
| `retryInterval` | — | `500` | Milliseconds between retries |

**Error handling**
```typescript
import { AgenticHubError } from '@agentichub/sdk';

try {
  const messages = await hub.renderPrompt('my-prompt', 'production', vars);
} catch (err) {
  if (err instanceof AgenticHubError) {
    if (err.code === 'MISSING_VARIABLES') {
      console.error('Missing template variables:', err.body.missing);
    } else if (err.code === 'NETWORK_ERROR') {
      console.error('AgenticHub API unreachable — check AGENTICHUB_BASE_URL');
    } else {
      console.error(`AgenticHub API error ${err.statusCode}`);
    }
  }
  throw err;
}
```

**Caching behaviour**

The SDK maintains an in-process LRU cache (default: 500 entries, 60s TTL). On a cache hit, the cached value is returned immediately; if the entry is stale, a background refresh is triggered and the stale value is served in the meantime. If the API is unreachable and a stale entry exists, the stale value is served silently. If the API is unreachable and the cache is cold, the SDK throws after exhausting retries.

Create one `AgenticHub` instance at process startup and reuse it — the cache is a module-level singleton and `maxCacheSize` is set by the first constructor call.

## Notes / Constraints

- **Node.js 18+ required** — uses native `fetch`. No polyfill, no `node-fetch`.
- **Cache singleton:** `maxCacheSize` is set by the first constructor call. Creating multiple instances with conflicting config is unsupported. Instantiate once.
- **Variables are not cache-keyed.** The cache key is `apiKey:name:alias`. If variable-sensitive caching becomes necessary later, hash variables into the key at that point.
- **No browser support.** LRU cache and fetch polyfills for browser environments are deferred.
- **No Redis caching.** In-memory LRU is sufficient for dogfood. Redis is a later upgrade.
- **No fetch-by-version-number.** Alias-based fetch only (`production`, `staging`, or any custom alias). Version-number fetch is deferred.
- **No nunjucks dependency.** All Jinja2 rendering is server-side; the SDK never touches template strings.
