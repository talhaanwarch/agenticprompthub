Phase 1 — Prompt Management

Build Decomposition Flowchart
Status: Approved
Builder: Solo + AI-paired
Cadence: 1–2 days per part
Total: ~2–3 weeks
Parts: 11 (6 backend + 5 frontend)
How Might We

A solo developer builds Phase 1 as backend-first, 1–2 day chunked parts — each part produces a tested, documented API — so the frontend is built once against a stable, complete contract.

Backend Track  ·  Parts 1–6
Backend Part 1
Schema + Auth API
Schema
users teams team_members api_keys
Endpoints
POST /auth/signup POST /auth/login POST /auth/logout GET /auth/me POST /api-keys DELETE /api-keys/:id
Done when: Signup, login, logout, and /me all work via curl/Postman. A personal API key can be generated and revoked.
Backend Part 2
Prompt CRUD + Versioning API
Schema
prompts prompt_versions (JSONB) prompt_aliases
Endpoints
POST /prompts GET /prompts PATCH /prompts/:id DELETE /prompts/:id POST /prompts/:id/versions GET /prompts/:id/versions POST /prompts/:name/:alias/render
Done when: A Jinja2 prompt can be committed, promoted to production, and rendered — returning a correct OpenAI-compatible messages array.
Backend Part 3
Versioning + Env Aliases API
Endpoints
GET /prompts/:id/versions/:v GET /prompts/:id/aliases POST /prompts/:id/aliases/:alias/promote
Auto-created aliases
production staging
Done when: A prompt can be saved (version created), promoted to production, and rolled back to a prior version — all via API.
Backend Part 4
SDK (TypeScript)
Package
@agenticprompthub/sdk Node.js only lru-cache stale-while-revalidate
API surface
new AgenticHub(config) renderPrompt(name, alias, vars)
Config options
apiKey / baseUrl cacheTtl (60s) maxCacheSize (500) maxRetries (1) retryInterval (500ms)
Done when: A Node.js script calls renderPrompt("my-prompt", "production", { name: "Alice" }) and receives a correct OpenAI-compatible messages array.
Backend Part 5
Diff + Audit Trail + Export/Import
Endpoints
GET /prompts/:id/versions/diff?from=v1&to=v2 GET /prompts/:id/audit GET /prompts/:id/versions/:v/export POST /prompts/import
Audit events captured
version committed alias promoted alias rolled back prompt created/deleted/renamed API key generated/revoked
Done when: Diff returns a unified diff string. Audit log shows full history. A prompt exported as JSON can be re-imported as a new prompt.
Backend Part 6
Teams + RBAC API
Schema
team_invites team_member_roles
Roles
owner admin editor viewer
Endpoints
POST /teams/:id/invites POST /teams/invites/:token/accept GET /teams/:id/members PATCH /teams/:id/members/:userId/roles DELETE /teams/:id/members/:userId POST /teams/:id/api-keys
Done when: Owner vs. Viewer demo — Viewer cannot commit versions or promote aliases. Invite link generates, is accepted, new member appears with correct roles.

Definition of Done — Phase 1 Complete

A developer fetches a production alias prompt at runtime via the SDK, edits the content in the UI, promotes the new version to production, and the next SDK fetch returns the updated content — without touching or deploying application code.

Explicitly Deferred from Phase 1

Multi-member teams (until B6) LLM calls / playground (Phase 2) Org layer OAuth / SSO Redis caching in SDK Import from PromptLayer / Langfuse Complex Jinja template logic Webhooks More than 2 env aliases Fetch by version number in SDK Per-prompt ACLs reviewer role
